# Тестирование `:crypto`

Тесты — часть «честности» открытого крипто-ядра: они позволяют любому независимо
**воспроизвести** проверки, а не верить на слово. Здесь описано, что покрыто, как это
запустить и что осознанно не автоматизируется.

## Принцип: результат воспроизводится запуском, а не файлом

Мы **намеренно не храним в репозитории статический отчёт** о прохождении (он мгновенно
устаревает и вводит в заблуждение). Источник истины — сам прогон. Машинные отчёты
(JUnit XML/HTML) пишутся в `crypto/build/` и исключены из git (`.gitignore`). Чтобы
получить актуальный результат — выполните команды ниже.

## Запуск

```bash
# Модульные (JVM, без устройства) — основной набор:
./gradlew :crypto:test

# Инструментальные (нужно подключённое устройство/эмулятор, экран разблокирован):
./gradlew :crypto:connectedAndroidTest
```

Зависимостей сверх Gradle не требуется: тесты — на стандартном JUnit4; Argon2 берётся из
BouncyCastle (та же зависимость, что и в продакшне).

## Объём

Набор содержит **73 проверки** — **66 модульных** (`src/test`, JVM) и **7 инструментальных**
(`src/androidTest`, реальный Android Keystore). Тесты сгруппированы по проверяемому
**свойству**, чтобы стороннему читателю было видно, что именно гарантируется.

## Карта покрытия по свойствам

| Свойство | Где проверяется | Что доказывается |
|---|---|---|
| **Round-trip + эталоны (KAT)** | `RoundTripTest`, `KnownAnswerTest` (+ аудио в `KeyManagerInstrumentedTest`) | шифр→дешифр возвращает данные **побайтово** (пустые, 1 MiB, все значения байта, кириллица/эмодзи); примитивы совпадают с **официальными векторами**: HKDF-SHA256 — RFC 5869 (TC1–3), Argon2id — RFC 9106 §5.3 |
| **Целостность / тамперинг** | `AeadIntegrityTest`, `BackupIntegrityTest` (+ аудио-тамперинг в инстр.) | любая модификация (флип бита в шифртексте/теге/IV/версии/соли/nonce, усечение, чужой ключ/версия) → расшифровка возвращает `null`, без частичной выдачи; фаззинг 5000 случайных входов не роняет `decrypt`; абсурдные KDF-параметры из заголовка отсекаются до запуска Argon2 |
| **Уникальность nonce/IV** | `NonceUniquenessTest` | IV/nonce уникальны на 10000 шифрований одних и тех же данных; две независимые «партии» не пересекаются → это **не счётчик от нуля**, а криптослучайность |
| **KDF и PIN** | `PinHasherTest`, `KnownAnswerTest` (Argon2id), `SecureConstructionGuardTest` (+ маппинг в инстр.) | верный PIN принимается, неверный/пустой отвергается, соль работает; параметры Argon2id (32 MiB, t=2) подтверждены **пересчётом**; сравнение — constant-time; аннулирование Keystore-ключа → `KeyInvalidatedException` |
| **Гигиена и безопасная конструкция** | `SourceHygieneTest`, `SecureConstructionGuardTest` | публикуемый `src/main` без NUL-байтов, захардкоженных секретов и личных данных; в коде сохранены безопасные конструкции (constant-time сравнение, затирание буферов в `finally`, только `SecureRandom`) |

## Тест-классы

**Модульные** (`src/test/java/app/arxnotes/core/crypto/`):

| Класс | Тестов | Кратко |
|---|--:|---|
| `RoundTripTest` | 11 | побайтовый round-trip AeadBlob/BackupCrypto + инварианты HKDF (вкл. отказ при недопустимой длине) |
| `KnownAnswerTest` | 9 | RFC 5869 (TC1–3) и RFC 9106 §5.3; детерминизм/чувствительность Argon2id |
| `AeadIntegrityTest` | 10 | тамперинг/усечение/неверный ключ AeadBlob → `null` |
| `AeadFuzzTest` | 3 | фазз `AeadBlob.open` на произвольных/граничных байтах → всегда `null`, без throw/hang |
| `BackupIntegrityTest` | 15 | тамперинг + фаззинг BackupCrypto; отсечка DoS-параметров (memory/iterations/parallelism, нулевые) |
| `NonceUniquenessTest` | 3 | уникальность IV/nonce; не счётчик от нуля |
| `PinHasherTest` | 8 | корректность PIN + KAT параметров Argon2id пересчётом |
| `SecureConstructionGuardTest` | 3 | стражи исходника: constant-time, затирание в `finally`, только SecureRandom |
| `SourceHygieneTest` | 4 | `src/main` без NUL/секретов/личных данных |

`TestSupport.kt` — общие **синтетические** фикстуры (тестовый AES-ключ, hex, bit-flip,
помеченные как тестовые пароль/PIN). Реальных секретов в тестах нет.

**Инструментальные** (`src/androidTest/java/app/arxnotes/core/crypto/`):

| Класс | Тестов | Кратко |
|---|--:|---|
| `KeyManagerInstrumentedTest` | 7 | мастер-ключ и HKDF-подключи (детерминизм, доменное разделение, персистентность); round-trip/тамперинг и большие/юникод-данные `AudioCrypto`; маппинг аннулирования ключа → `KeyInvalidatedException`; повреждённый блоб мастера на разблокированном устройстве → `Corrupt key blob` |

## Эталонные векторы

- **HKDF-SHA256** — RFC 5869, Appendix A, Test Case 1–3 (точные опубликованные значения).
  TC3 (пустая соль) прогоняется через путь `salt = null`, который по RFC §2.2 эквивалентен
  «HashLen нулей» и даёт каноническое значение вектора.
- **Argon2id** — RFC 9106, §5.3 (точные байты вектора, с `secret` и `associated data`).
  Проверяется тот же движок (BouncyCastle), что используется в продакшне.

## Что НЕ покрыто автотестами и почему

- **«AEAD настроек»** — такого пути нет: настройки хранятся в открытом DataStore по дизайну
  (см. `THREAT_MODEL.md`), отдельного шифрования настроек в ядре не существует. Не пробел.
- **Реальное аннулирование Keystore-ключа операционной системой** — ОС делает это только при
  смене защиты экрана, программного API нет. Автоматизирована **наша логика** преобразования
  исключения; сам факт ОС-аннулирования проверяется вручную (см. ниже).
- **Постоянство времени сравнения, фактическое зануление байтов в RAM, качество ГСЧ** —
  напрямую из публичного API не наблюдаемы. Свойства обеспечиваются конструкцией
  (`MessageDigest.isEqual`, `fill(...)` в `finally`, `SecureRandom`); `SecureConstructionGuardTest`
  фиксирует эти конструкции на уровне исходника и ловит регрессию, но не доказывает
  рантайм-поведение. Это честно отражено в KDoc соответствующих тестов.

## Ручная проверка (что нельзя автоматизировать)

Аннулирование ключа Android Keystore система выполняет только при смене защиты экрана —
программного API для этого нет, поэтому путь проверяется вручную:

1. Включите блокировку приложения с PIN и создайте пару заметок.
2. В системных настройках телефона смените или удалите PIN/графический ключ экрана.
3. Откройте приложение: ключ-обёртка аннулирована ОС → ожидается состояние
   «данные невосстановимы» (`KeyInvalidatedException`), а **не** «файл повреждён».

Преобразование системного `KeyPermanentlyInvalidatedException` в `KeyInvalidatedException`
покрыто автотестом (`KeyManagerInstrumentedTest.mapKeystoreInvalidation_*`); ручная проверка
подтверждает лишь сам факт ОС-аннулирования.
